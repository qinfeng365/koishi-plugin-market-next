# Changelog

本文件记录 `koishi-plugin-market-next` 已发布到 npm 的版本差异。

## Unreleased

暂无。

## 3.6.0-alpha.4

这是插件包（Plugin Bundle）能力的第四个 alpha 测试版，继续发布到 `alpha` 渠道，不替换 `latest`。

- 修复已安装插件包在插件市场点击“修改”时先进入普通安装弹窗的问题；插件包现在会直接打开专用成员安装/维护面板。
- 修复依赖管理页中插件包卡片仍按普通依赖展开版本操作的问题；单击插件包卡片或点击“管理”会直接进入插件包面板。
- 插件包入口识别补充 `market:package` keyword，避免只有显式插件包标记但包名未命中兜底规则时误走普通插件安装流程。

## 3.6.0-alpha.3

这是插件包（Plugin Bundle）能力的第三个 alpha 测试版，继续发布到 `alpha` 渠道，不替换 `latest`。

- 修复插件包安装确认弹窗在小屏、高缩放或成员较多时无法完整滚动的问题；弹窗现在挂载到 body，整体限制在视口内，并让内容区独立滚动。
- 优化插件包安装面板的信息密度，收紧成员卡片、统计区、说明文本和最终 diff 的间距，减少小屏设备上的视觉压迫。
- 移除插件市场、依赖管理、安装弹窗、插件包弹窗等区域突兀的白色滚动条轨道，改为透明轨道与更柔和的 thumb 样式。
- 插件包安装弹窗底部操作区固定在弹窗内，用户审查成员列表和配置预设后仍能直接执行安装或取消。
- 更新前端构建链路的 `esbuild` 到安全修复版，并调整 Console 前端构建目标，确保发布 workflow 的 audit 与 build 同时通过。

- 修复插件包卸载弹窗在依赖管理页中被分组滚动容器裁切的问题；卸载弹窗现在挂载到 body，并限制宽度与内容滚动区域。
- 修复插件包卸载弹窗成员操作按钮在窄屏或小容器内撑开布局的问题，成员处理选项改为响应式网格。

- 插件包卡片增加更明确的视觉区分，同时整理标题徽标与右侧“详情 / 修改”操作区的布局，避免互相重叠。

## 3.6.0-alpha.1

这是插件包（Plugin Bundle）能力的第一个 alpha 测试版，发布到 `alpha` 渠道，不替换 `latest`。

- 新增插件包（Plugin Bundle）第一阶段能力：支持通过 `koishi-plugin-pa-xxx` / `@scope/koishi-plugin-pa-xxx` 包名、`market:package` keyword 和 `koishi.bundle` 清单识别插件包。
- 插件市场卡片增加“插件包”标识，并支持 `is:bundle` / `not:bundle` 搜索过滤；未安装插件包点击安装时进入专用确认 GUI。
- 新增安全安装 GUI：展示成员清单、required / optional、版本范围、安装状态、描述、作者/维护者、下载量、风险标签、完整预设配置和最终 diff。
- 插件包预设配置不会静默注入；用户可逐个选择创建停用配置、使用预设配置、跳过配置或移动已有配置到插件包分组，敏感字段会在确认面板中提示。
- 插件包成员默认会在插件包分组下创建停用配置；如果成员已存在根层级或其他分组配置，会保留原配置，并额外创建包分组副本，除非用户显式选择移动已有配置。
- 新增 `market/install-bundle` Console 事件；后端会重新读取 npm 元数据并校验远端 `koishi.bundle`，拒绝缺失 version、自引用、非法包名和直接循环。
- 插件包安装成功后写入隐藏的本地归属记录 `market.bundleRecords`，依赖管理卡片会显示成员来源为对应插件包。
- 依赖管理页、市场“修改”面板、插件配置详情页和包分组右键菜单统一使用插件包卸载弹窗。
- 插件包卸载弹窗支持逐成员选择“只清包内配置 / 卸载依赖并清配置 / 保留”；如果成员存在包分组外配置，默认只清包内配置并禁止直接卸载该成员依赖。
- 批量模式下会同时暂存插件包自身、被选择卸载的成员依赖以及只清理包内配置的成员列表，应用更改后统一执行。
- 配置页包分组右键“卸载插件包”会按包名或本地归属记录识别对应插件包；插件包自身不再被归入“已下载未配置”。
- `npm run check:package` 增加插件包发布校验，检查清单结构、semver range、重复成员、plugin 键冲突风险和本地可解析成员的直接循环。
- README 新增“插件包 / Plugin Bundle”发布指南，说明 npm 小写命名、`market:package`、`koishi.bundle` v1、安装安全策略和当前阶段限制。

## 3.5.6

这是从 `3.5.5` 到 `3.5.6` 的正式版汇总，整合了 `3.5.6-alpha.0` 至 `3.5.6-alpha.6` 的测试内容，并补充了正式版前的精致模式与路由持久化优化。

- 依赖管理页从全量表格改为按状态分组的卡片工作台，增加搜索、状态筛选、摘要统计、固定底部“待应用”操作条和 `Ctrl+K` / `Cmd+K` 搜索聚焦。
- 依赖卡片改为紧凑 row-card：展示短名、完整包名 tooltip、当前/最新/目标版本、配置状态、来源类型、异常原因和主要操作；普通已安装项降低视觉权重，待应用、可更新、未配置、异常等状态保持突出。
- 增加“已下载未配置”分组和“添加配置”入口，已安装但插件配置页尚未出现的插件可以直接补齐停用配置。
- 依赖卡片支持单击展开 / 收起操作区，并增加明确的“卸载”按钮；卸载、安装、更新等变更会保留在待应用分组中，直到用户确认或取消。
- 增加依赖更新提示策略：可填写不检测更新的依赖名单，可按时长和连续版本数忽略单个更新，可恢复提示；已忽略更新进入默认折叠分组。
- 增加手动开启的预发布版本过滤，开启后 `alpha`、`beta`、`rc` 等 prerelease 版本不会被视为可更新目标，“全部更新”也会避开这些版本。
- 依赖版本刷新优先返回本地依赖快照，不再等待整轮 npm 元数据刷新完成；最新版本信息由后台刷新后推送到 Console。
- npm 元数据路由增加主源优先的 route probe：每轮刷新先用代表包探测可用源，后续包复用同一轮选中的 `metadataEndpoint`，减少弱网下每个包重复试多个 npm 源的等待放大。
- 普通包元数据请求也支持慢源接管：首选 npm 源失败或明显超时时，备用源可在同一个请求内胜出并更新当前 `metadataEndpoint`。
- npm registry 路由评分按成功率、失败原因、平均耗时和最近成功动态调整；超时和网络失败重惩罚，`not-found` / 镜像未同步轻惩罚，避免真实包不存在时污染整个源评分。
- market 索引路由和 npm registry 路由的慢源接管阈值从 1.5 秒降到 800ms，弱网下更早启动备用源竞速。
- market route stats 与 npm registry route stats 增加磁盘持久化，重启后保留历史平均延迟、最近成功和评分，减少每次冷启动重新试源的等待。
- 市场索引缓存增加 30 天过期清理；写盘时自动裁剪过期条目，避免多源缓存长期增长。
- npm 包元数据增加 5 分钟 `not-found` 负缓存，已确认不存在的包不会在同一段时间内被反复请求。
- 市场页和依赖管理页增加 `frontendMode` 配置：默认“性能模式”保持低动效和高密度；手动切到“精致模式”后启用更细腻的卡片、工具栏、分组、搜索框、提示条、滚动条和入场动效。
- 精致模式下市场卡片增加交错入场动画、柔和玻璃质感、hover 光晕和图标动效；虚拟滚动列表在首屏稳定后停止重复播放入场动画，避免高速滚动闪烁。
- 精致模式下依赖管理页增加毛玻璃 toolbar、分组标题渐变、状态色细节、卡片 hover 层次、搜索框焦点态和底部待应用操作条强化。
- 市场页搜索改进为大小写不敏感，并兼容 manifest 多语言描述；市场页和依赖管理页均支持 `Ctrl+K` / `Cmd+K` 聚焦搜索。
- 市场缓存 / stale 提示会在 `search.logLevel: silent` 时隐藏，符合静默模式预期。
- 插件配置详情页增加 market-next 的“卸载插件包”入口，并将 config 原生删除确认文案改为“删除插件配置”，避免把删除配置误解为卸载 npm 依赖。
- npm / Koishi 插件市场展示描述改为 market-next 专属定位：下一代 Koishi 插件市场与依赖管理中心。

## 3.5.6-alpha.2

这是面向弱网用户的第三个 alpha 测试版，继续发布到 `alpha` 渠道，不替换 `latest`。

- 依赖管理页优先返回本地依赖快照，不再等待整轮 npm 元数据刷新完成；最新版本信息改为后台刷新后再推送到 Console。
- 依赖刷新按钮反馈改为“依赖版本刷新已开始”，匹配后台刷新语义，避免用户误以为所有远端版本已经同步完成。
- 普通包元数据请求不再死磕已选中的 `metadataEndpoint`：首选源慢或失败时会在同一个包请求内启动备用源 race，备用源胜出后立即更新 `metadataEndpoint`，避免一直优先请求 `npmmirror` 等已变慢的源。
- 后台依赖元数据刷新失败时会记录 warning 并清理任务句柄，避免异常状态卡住下一轮刷新。
- `audit:high` 与 `audit:package` 统一排除 peer runtime，避免 npm quick audit 把可选的 Koishi / ChatLuna 宿主依赖树误判为 invalid package tree。

## 3.5.6-alpha.1

这是面向弱网用户的第二个 alpha 测试版，继续发布到 `alpha` 渠道，不替换 `latest`。

- 重载 market-next 配置后，已安装插件配置补齐扫描改为延迟后台执行，不再阻塞插件 `ready`；扫描前会先判断是否已有配置，只对缺配置插件调用 `config/request-runtime`。
- 批量补齐插件配置时改为一次扫描、一次写入配置文件，并在长列表中定期让出事件循环，降低 100+ 插件环境下的卡顿。
- npm 元数据 route probe 从“全源立即并发竞速”改为“主源优先、主源慢或失败后备用源接管”；备用 npm 源按成功率、失败惩罚、平均耗时和最近成功排序，与市场索引自动路由策略保持一致。
- npm 元数据路由增加动态 fallback delay：主源连续失败或历史平均耗时过高时缩短备用源启动等待，主源健康时仍保持默认 1.5 秒观察窗口。
- npm 元数据路由按失败原因调整降权幅度：超时和网络失败重惩罚，HTTP 错误中等惩罚，`not-found` / 镜像未同步轻惩罚，避免真实包不存在时错误拉低所有源。
- 普通包元数据请求也使用 route score 排序；当当前 `metadataEndpoint` 连续失败且分数低于主源时，会自动降级回主源优先。
- 检测到 `registry.endpoint` 变化时清空旧 npm 路由评分，避免旧主源/备用源历史污染新配置。

## 3.5.6-alpha.0

这是面向弱网用户的 alpha 测试版，不会替换 `latest` 渠道。

- route probe 结果只保存在当前进程和当前刷新序列中，不写入用户配置，也不改变安装命令最终使用的 npm registry。
- 依赖刷新日志补充 `npm registry route probe started/selected/failed`、`refresh dependency metadata route ready` 和完成时实际 `registry`，便于在日志页判断慢在源选择、单包请求、镜像未同步还是网络超时。
